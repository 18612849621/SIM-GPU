# 阶段 1.2：8×32 位通用寄存器文件

PC 只保存一个特殊值——下一条指令的位置。处理器还需要一组可以保存操作数和计算结果的位置，例如以后执行 `ADD r2, r3, r4` 时，要同时读出 `r3` 与 `r4`，并在稍后的时钟边沿把结果写入 `r2`。这组位置就是寄存器文件（register file）。

本阶段只实现独立的寄存器文件；不接入顶层、不定义指令，也不实现 ALU。这样我们一次只学习一种新硬件机制。

## 1. 8×32 位是什么意思

寄存器文件包含 8 个独立的 32 位寄存器：

```text
r0  [31:0]
r1  [31:0]
r2  [31:0]
r3  [31:0]
r4  [31:0]
r5  [31:0]
r6  [31:0]
r7  [31:0]
```

- `8`：可选择 `r0` 到 `r7` 共 8 个位置。
- `32 位`：每个位置能保存一个 32 位数。
- 选择 8 个位置需要 `3` 位地址，因为 `2³ = 8`：`000` 选 `r0`，`111` 选 `r7`。

当前所有寄存器都可以读写。此时不规定 `r0` 必须为零；那是某些 ISA 的额外规则，等定义指令集时再决定。

## 2. 模块端口

`src/regfile.sv` 的接口：

```text
                    ┌──────────────────────┐
clk ───────────────►│                      │
rst ───────────────►│                      │
write_enable ──────►│                      │
write_addr[2:0] ───►│       regfile        │───► read_data_a[31:0]
write_data[31:0] ──►│       8 × 32 位      │───► read_data_b[31:0]
read_addr_a[2:0] ──►│                      │
read_addr_b[2:0] ──►│                      │
                    └──────────────────────┘
```

| 端口 | 方向 | 含义 |
| --- | --- | --- |
| `clk` | 输入 | 写入和同步复位只在上升沿生效 |
| `rst` | 输入 | 高有效同步复位 |
| `write_enable` | 输入 | 为 `1` 时允许写入 |
| `write_addr[2:0]` | 输入 | 要写的寄存器编号 |
| `write_data[31:0]` | 输入 | 要保存的 32 位数据 |
| `read_addr_a/b[2:0]` | 输入 | 两个独立读端口要选择的编号 |
| `read_data_a/b[31:0]` | 输出 | 两个读端口当前读出的值 |

一个写端口表示每个时钟周期最多写一个寄存器；两个读端口表示可在同一时刻读两个寄存器。这正好对应大多数双操作数 ALU 指令的需要。

## 3. 状态、同步写入与组合读取

内部数组 `registers[0:7]` 是保存状态的硬件。它与 PC 一样，只能由时钟上升沿更新：

```systemverilog
always_ff @(posedge clk) begin
    if (rst) begin
        // 将 r0 到 r7 清零
    end else if (write_enable) begin
        registers[write_addr] <= write_data;
    end
end
```

上升沿时的规则：

| `rst` | `write_enable` | 上升沿后的寄存器状态 |
| ---: | ---: | --- |
| `1` | 任意 | `r0` 到 `r7` 全部为 `0` |
| `0` | `1` | 只有 `write_addr` 指定的寄存器变为 `write_data` |
| `0` | `0` | 所有寄存器保持原值 |

复位优先于写入：即使 `rst=1` 与 `write_enable=1` 同时出现，上升沿后仍是全部清零。

读端没有时钟：

```systemverilog
assign read_data_a = registers[read_addr_a];
assign read_data_b = registers[read_addr_b];
```

这叫**组合读取**。例如 `read_addr_a` 从 `3` 改为 `6` 后，`read_data_a` 会直接反映 `r6` 当前保存的值；它不必等到下一个上升沿。读操作不修改任何寄存器。

## 4. 写入时序

测试在下降沿准备好输入，使它们在下一个上升沿前稳定。写入在上升沿提交；此后组合读端立即能选择到新保存的值。

```text
时间 →                t0              t1              t2

clk = 1                             ┌────┐         ┌────┐
clk = 0  ───────────────────────────┘    └─────────┘    └──
                                        ↑

write_enable = 1  ─────────────────────────────────────────
write_addr        ──────────────── r2 ──────────────────────
write_data        ─────────── 0x12345678 ────────────────────
read_addr_a       ──────────────── r2 ──────────────────────
read_data_a       ──────────────── 旧值 ─── 0x12345678 ──────
```

这里 `t0` 的下降沿后，测试设置写地址、写数据和读地址。`t1` 的上升沿发生后，`r2` 保存 `0x12345678`。因此在 `ReadOnly()` 阶段读取 `read_data_a`，会得到新值。若读地址和写地址不同，读端则继续显示它所选择的另一个寄存器。

## 5. 为什么单独测试

`test/test_regfile.py` 直接把 `regfile` 作为仿真顶层，分别验证：

1. 同步复位让全部 8 个寄存器清零；
2. `write_enable=0` 时，写地址和写数据不会改变状态；
3. 分别写入 `r2` 和 `r6`，两个读端口可同时读到它们；
4. 写 `r2` 不会影响 `r6`；
5. 同址读写时，上升沿后的 `ReadOnly()` 能读到新值；
6. 再次复位会覆盖已保存的非零数据和同一拍的写请求。

测试仍采用阶段 0 的时序纪律：在 `FallingEdge()` 后修改输入，在 `RisingEdge()` 后等待 `ReadOnly()`，再检查同步逻辑更新后的稳定结果。

## 6. 运行

只运行寄存器文件测试：

```bash
source .venv/bin/activate
make test_regfile
```

运行项目全部测试：

```bash
make test
```

`Makefile` 为 `gpu_top`、`pc` 和 `regfile` 使用各自的 `sim_build/` 目录，避免不同硬件顶层错误复用编译产物。
