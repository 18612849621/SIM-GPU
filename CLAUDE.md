# SIM-GPU 开发计划

## 目标

从零实现一个可仿真的教学型 GPU。优先建立清晰、可验证的执行模型，再逐步加入 GPU 特性；每个阶段都必须有可重复的自动化测试。

第一版的最终目标是让多个线程执行同一程序，完成向量加法：

```text
output[thread_id] = input_a[thread_id] + input_b[thread_id]
```

## 开发原则

- 每次只引入一个概念或硬件机制，并为它增加测试。
- 先实现正确的多周期 RTL，再考虑流水化、缓存和性能优化。
- 在顶层集成前，为寄存器、ALU、译码器和 PC 等模块写独立测试。
- 指令格式、端口时序和内存握手协议必须先写清楚，再实现 RTL。
- 所有状态由时钟驱动；组合逻辑不得意外产生锁存器。
- 每个阶段通过后再进入下个阶段，不保留已知失败的测试。

## 分阶段路线

### 阶段 0：项目骨架与仿真环境

目标：建立可执行、可验证的 SystemVerilog 项目。

- 添加 `Makefile`，使用 Icarus Verilog 运行仿真。
- 添加最小 SystemVerilog testbench 或 Cocotb 测试框架。
- 创建 `src/` 与 `test/` 目录。
- 增加一个时钟、复位和断言都能工作的空顶层模块。

完成条件：`make test` 可以稳定执行并通过一个复位测试。

### 阶段 1：单线程最小处理器

目标：实现不含 GPU 并行机制的顺序处理器，先掌握指令执行闭环。

硬件：

- 一个程序计数器（PC）。
- 8 个 32 位通用寄存器。
- 指令存储器和数据存储器模型。
- 一个 ALU。
- 一个多周期控制状态机。

最小指令集：

```text
CONST rd, imm
ADD   rd, rs1, rs2
LDR   rd, [rs1 + imm]
STR   rs, [rs1 + imm]
HALT
```

完成条件：能运行一个程序，读取两个内存值、相加、写回内存，然后停机。

### 阶段 2：完善单线程执行模型

目标：理解指令译码、条件码和控制流。

- 明确定义 16 位或 32 位指令格式。
- 实现 `SUB`、`MUL`、比较和条件分支。
- 为分支定义条件码或比较结果。
- 在波形中确认每条指令的 fetch、decode、execute、memory、writeback 时序。

完成条件：能运行带循环的数组求和程序。

### 阶段 3：单 core 多线程 SIMD 执行

目标：把单线程处理器扩展为一个可执行 thread block 的 core。

- 固定一个 block 内的线程数，初始设为 4。
- 每个线程拥有独立 PC、寄存器和 active 状态。
- block 内线程共享取指、译码和控制状态机。
- 增加只读线程编号寄存器 `thread_id`。
- 对所有活跃线程执行相同指令；暂不支持线程分支分歧。

完成条件：4 个线程各自完成一次向量加法元素计算。

### 阶段 4：访存与同步

目标：让多线程能正确地使用外部数据存储器。

- 定义请求/响应式内存接口：地址、读写类型、写数据、有效位、ready/response。
- 实现每线程 Load/Store 请求状态。
- 控制器必须等待所有活跃线程的当前访存完成后，才推进下一条指令。
- 先采用简单、确定性的串行仲裁。

完成条件：多线程向量加法通过不同内存响应延迟下的测试。

### 阶段 5：多 core 与 block 调度

目标：让一个 GPU 顶层调度多个 block。

- 实现配置寄存器，至少包含总线程数和启动信号。
- 实现 dispatcher，将线程按固定 block 大小派发给空闲 core。
- 初始配置为 2 个 core、每 block 4 个线程。
- 实现所有 block 完成后的 `done` 信号。
- 为多个 core 实现数据内存请求仲裁。

完成条件：线程数不是 block 大小整数倍时，仍能正确完成向量加法。

### 阶段 6：可观测性与扩展

目标：使设计易于调试，并在稳定后选择性扩展。

- 为关键状态添加波形检查和断言。
- 补充 ISA、内存协议、block 生命周期和时序文档。
- 增加矩阵加法与小矩阵乘法程序。
- 只在基线稳定后再评估：分支分歧、流水化、shared memory、cache、更多 ALU 指令。

## 当前进度

当前正在阶段 1 构建最小单线程处理器的基础状态：已实现 32 位 PC 的独立模块、顶层集成和测试。下一步实现通用寄存器文件。
